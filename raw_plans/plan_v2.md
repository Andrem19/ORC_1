# План дальнейших исследований для v2

## Статус после plan_v1

**База, которую не трогаем:** `active-signal-v1@1`.

Эта база остается фиксированной опорной системой. Новый цикл не занимается ее ретюнингом, не подбирает часы, дни, пороги, исполнение или косметические вариации старых правил.

### Что показал plan_v1

1. **Простой event timing сам по себе не работает.**
   Гипотезы вида `post-funding` и `pre-expiry` с одним слабым подтверждением породили слишком много входов и умерли в комиссиях.
2. **Одномерные сигналы низкой плотности нам не нужны.**
   Новая идея должна приходить не как один фильтр, а как более строгая архитектура из нескольких причин для входа.
3. **Главная ценность нового слоя — не обязательно standalone-сигнал.**
   По всей истории исследований новый слой чаще всего полезен как conditioner, router или specialist-sleeve для missing regime, а не как грубый массовый standalone-вход.
4. **Нельзя тратить цикл на идеи, которые не готовы инфраструктурно.**
   Если для семьи не готов data contract, custom feature или cross-symbol dataset, это сначала отдельная техническая задача, а не исследовательское утверждение.

### Главный разворот v2

**Plan v2 меняет не только семейства гипотез, но и саму архитектуру поиска.**

Вместо вопроса:

> «Есть ли красивый новый standalone-сигнал?»

теперь основной вопрос такой:

> «Есть ли новый слой, который на участках, где база молчит, дает качественно новые входы и потом сохраняет ценность при честной интеграции?»

Из этого следуют три жестких правила:

- сначала исследуем **пространство, где база не входит**;
- сначала строим **strict specialist candidates**, а не массовые слабые входы;
- сначала проверяем **ортогональность по конструкции**, а не только постфактум через overlap.

---

## Главная цель цикла

Найти один или несколько **новых ортогональных specialist-сигналов или conditioning-слоев**, которые:

- работают прежде всего там, где база `active-signal-v1@1` не торгует;
- используют новый слой информации или новый тип маршрутизации;
- не являются переименованной вариацией старых `hour`, `dow`, `cl_*`, `rsi`, `iv_est`, `Monday/Thursday` паттернов;
- проходят window stability;
- при интеграции улучшают общую сборку и доказывают, что их вклад идет через **новые входы**, а не через скрытый дубляж базы.

---

## Основные выводы, которые обязательно учитывать в v2

### Что больше не делать

1. **Не повторять H1/H2 в прежнем виде.**
   Сырые event-entry конструкции вида «событие плюс один фильтр» признаны тупиком.
2. **Не искать массовые low-conviction сигналы.**
   Если кандидат быстро выходит в сотни сделок без явного edge до комиссий, это почти наверняка шум.
3. **Не начинать с модели там, где еще не исчерпаны более простые structured hypotheses.**
4. **Не оценивать новую идею только на full-history standalone-графике.**
5. **Не принимать кандидата, если его ценность исчезает в режиме "candidate fires only when base is absent".**

### Что теперь делать обязательно

1. **Строить карту "base absent surface".**
2. **Искать missing regime не глобально, а внутри участков, где база молчит.**
3. **Строить кандидатов минимум из трех осмысленных компонентов:**
   - новый слой;
   - подтверждение режима или структуры;
   - ограничитель плотности входов.
4. **Сначала проверять additive mode by construction:** кандидат имеет право входить только когда база не вошла на этом anchor.
5. **Проверять два режима интеграции:**
   - strict additive: кандидат запрещен, если база уже вошла;
   - permissive integrated: обычная интеграция.

Если permissive режим красивый, а strict additive резко слабый, это тревожный признак скрытого каннибализма.

# ЭТАП 2. Wave A — low-friction новые слои на уже доступных данных

## Цель

Сначала выбрать гипотезы, которые можно построить **без тяжелой внешней инфраструктуры**, используя уже доступные 1h данные и минимальное число новых custom features.

## Приоритет v2

### Семейство A1. IV dynamics from existing `iv_est_1`

Не использовать `iv_est` как статический bucket. Использовать **динамику** implied vol:

- рост за короткий отрезок;
- падение за короткий отрезок;
- ускорение роста или падения;
- смену знака динамики;
- растянутость текущего IV относительно недавней истории;
- сочетание шока и последующего затухания.

### Семейство A2. Event × state interactions

Не raw event timing. Только structured interactions:

- funding/expiry distance в сочетании с IV dynamics;
- funding/expiry distance в сочетании с classifier state;
- post-event continuation/failure только при явном state confirmation.

### Семейство A3. Classifier-transition specialists

Не статический `cl_*`, а **переход режима**:

- один класс на предыдущем окне и другой сейчас;
- переход из слабого режима в сильный;
- переход из перегретого режима в сжатый;
- переход, который исторически чаще возникает именно там, где база молчит.

### Семейство A4. Compression-to-release without model

До моделей проверить, можно ли построить простые причинные non-model признаки:

- сжатие диапазона;
- низкая реализованная подвижность;
- спад активности;
- затем release с подтверждением по режиму.

## Что сделать

1. Для каждого семейства описать hypothesis contract:
   - missing regime;
   - почему новый слой не дубль базы;
   - почему он должен сработать именно на `base_absent` surface.
2. Для A1 и A2 опубликовать необходимые custom features.
3. Для A3 и A4 проверить, какие признаки уже есть, а какие нужно добавить.
4. Не делать сразу много кандидатов. Выбрать максимум **6–8 hypothesis prototypes**.

## Критерий завершения этапа

Есть небольшой, но качественный список hypotheses wave A, все они data-ready или почти data-ready.

# ЭТАП 5. Построение strict specialist prototypes

## Цель

Перевести выжившие идеи в реальные сигналы, но не повторить ошибку H1/H2.

## Новое правило v2

Каждый prototype должен иметь **минимум три логических слоя**:

1. **новый слой** — IV dynamics, transition, event-state interaction или compression layer;
2. **подтверждение режима** — classifier state, RSI regime, context regime, event subspace;
3. **ограничитель плотности** — чтобы сигнал не становился массовым шумом.

### Пример правильной архитектуры

Не так:

> funding settlement недавно и RSI низкий

А так:

> funding settlement недавно, IV резко изменился, classifier перешел в конкретный режим, и все это находится на участке, где база historically absent.

## Что сделать

1. Для каждого кандидата построить 2–4 strict prototypes.
2. Не допускать широкие parameter sweeps.
3. Для каждого prototype заранее записать:
   - expected specialist regime;
   - why additive by construction;
   - expected trade density;
   - почему он не должен каннибализировать базу.
4. Если prototype снова выдает сотни слабых входов, его не тюнинговать, а отклонять.

## Критерий завершения этапа

Есть небольшой набор строгих specialist-прототипов, а не рыхлая сетка слабых фильтров.

# ЭТАП 8. Интеграция с базой в двух режимах

## Цель

Проверить не только то, красив ли кандидат рядом с базой, но и то, насколько он действительно additive.

## Два обязательных режима

### Режим 1. Strict additive

Кандидат запрещен на любом anchor, где база уже вошла.

Этот режим нужен как главный тест на реальные новые входы.

### Режим 2. Permissive integrated

Обычная интеграция без жесткого запрета, чтобы увидеть полный итоговый эффект.

## Что сделать

1. Для каждого финалиста прогнать strict additive integration.
2. Затем прогнать permissive integration.
3. Сравнить оба режима с неизменной базой.
4. Сохранить:
   - delta PnL;
   - delta trades;
   - delta PF;
   - delta DD;
   - weak-quarter repair;
   - разницу между strict и permissive режимами.

## Как трактовать результат

- **Если strict additive уже хорош** — сильный признак ортогональности.
- **Если strict слабый, а permissive сильный** — риск скрытого дубляжа.
- **Если оба слабые** — кандидат отклоняется.

## Критерий завершения этапа

Есть понятная картина: кто реально additive, а кто красив только за счет смешивания с базой.

# ЭТАП 9. Cannibalization, ownership и new-entry proof

## Цель

Формально доказать, что новый слой принес именно новые сделки и новую прибыль.

## Что сделать

Для каждого интеграционного кандидата посчитать:

1. overlap с базой;
2. число действительно новых сделок;
3. сделки базы, которые были вытеснены;
4. PnL именно новых сделок;
5. ownership прибыли;
6. coverage expansion — какие часы, состояния, event-подпространства теперь покрыты.

## Жесткие правила принятия

Кандидат не принимается, если:

- overlap высокий, а strict additive слабый;
- прибыли новых сделок мало;
- ownership показывает, что кандидат не является носителем прибыли;
- новый слой в основном перехватывает хорошие сделки базы.

## Критерий завершения этапа

Есть прямое доказательство, кто реально расширяет систему.
